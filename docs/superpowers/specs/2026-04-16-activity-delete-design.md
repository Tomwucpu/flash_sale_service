# 活动删除设计

## 背景

当前前后端已经支持活动创建、编辑、发布、下线和列表展示，但后台活动管理页还不支持删除活动。活动表 `activity_product` 已存在 `is_deleted` 字段，现有列表与详情查询也已经按 `is_deleted = 0` 过滤，因此最合适的方案是在现有状态流转之外补一条逻辑删除链路，而不是物理删除数据。

## 目标

- 后台支持删除活动。
- 删除仅允许发生在 `UNPUBLISHED` 或 `OFFLINE` 状态。
- 删除后活动不会出现在后台列表和详情查询结果中。
- 删除时同步清理 Redis 中可能残留的活动缓存和可见索引。
- 前端活动列表补充删除入口、确认提示和禁用态说明。

## 非目标

- 不实现物理删除。
- 不扩展新的活动状态。
- 不新增批量删除、回收站或恢复功能。
- 不改动活动详情页和编辑页的主流程。

## 方案选择

### 方案 A：逻辑删除

通过删除接口将 `activity_product.is_deleted` 置为 `1`，同时保留原有活动状态字段。所有现有查询已经过滤未删除记录，因此删除后活动会自然从列表和详情中消失。

优点：

- 与当前数据库结构和查询逻辑完全一致。
- 风险低，不会破坏后续审计或关联数据能力。
- 改动集中，前后端都只需增量扩展。

### 方案 B：物理删除

直接删除数据库记录。

不采用原因：

- 会破坏历史记录保留能力。
- 后续若挂接兑换码、订单等关联数据，清理成本和风险更高。

## 最终设计

采用方案 A。

### 后端

- 在 `ActivityAdminController` 新增 `DELETE /api/activities/{activityId}`。
- 在 `ActivityService` 新增删除方法：
  - 先按 `id + is_deleted = 0` 查询活动。
  - 仅当活动状态为 `UNPUBLISHED` 或 `OFFLINE` 时允许删除。
  - 将 `is_deleted` 置为 `1`，同时更新 `updatedBy`。
  - 调用 `ActivityCacheService.clear(activity)`，清理库存 Key、详情 Key 和可见活动索引。
- 对已发布活动删除返回 `400 INVALID_ARGUMENT`，错误文案明确为仅未发布或已下线活动允许删除。

### 前端

- 在 `frontend/src/api/activity.ts` 新增删除请求方法。
- 在后台活动列表页新增删除按钮和确认弹窗。
- 删除按钮规则：
  - `UNPUBLISHED` 和 `OFFLINE` 可点击。
  - `PUBLISHED` 禁用。
- 删除成功后刷新列表并提示成功。
- 操作提示文案同步更新，明确未发布和已下线活动都可删除。

## 数据与状态影响

- 删除不会新增新状态，仍沿用 `publishStatus`。
- 删除后的活动因 `is_deleted = 1` 不再被后台列表、详情和定时发布扫描命中。
- 即使是已下线活动，删除时仍统一调用缓存清理，保证 Redis 无残留。

## 错误处理

- 活动不存在或已被删除：返回 `活动不存在`。
- 活动状态为 `PUBLISHED`：返回 `仅未发布或已下线活动允许删除`。
- 前端沿用现有 `ApiClientError` 展示后端错误信息。

## 测试策略

### 后端

- 新增控制器测试，覆盖：
  - 未发布活动可以删除。
  - 已下线活动可以删除。
  - 已发布活动删除被拒绝。
  - 删除后列表不再返回该活动。
  - 删除时触发 Redis 缓存清理。

### 前端

- 优先保证类型检查和构建通过。
- 若列表页已有可复用测试基建，则补一条删除入口的交互测试；若当前仓库未形成页面测试基线，则至少验证删除 API 接入、按钮禁用和文案渲染逻辑无类型错误。

## 风险与边界

- 当前删除为逻辑删除，数据库里仍保留记录，后续若新增后台审计页面可以直接复用。
- `OFFLINE` 状态活动重复执行缓存清理是幂等的，可接受。
- 本次不处理删除后的关联兑换码清理，因为当前需求只要求活动管理删除，且 `redeem_code` 已有独立删除标记字段，后续如需联动可单独设计。
