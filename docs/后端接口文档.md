# 高并发秒杀兑换平台后端接口文档

> 文档版本：v1.5  
> 编写日期：2026-04-19  
> 适用范围：**仅包含当前代码中已经开发完成并可调用的业务接口**  
> 当前阶段：已完成 `Task 1`、`Task 2`、`Task 3`、`Task 4`、`Task 5`，并补充用户侧公开活动查询接口

## 1. 文档范围

当前仓库里已经落地的业务接口来自以下三个服务：

- `flash-sale-user-service`：用户注册、登录、当前用户查询、后台按 ID 查询用户
- `flash-sale-activity-service`：活动创建、编辑、详情、列表、发布、下线、删除、第三方兑换码导入、导入批次查询、导入失败明细查询、用户侧公开活动列表与详情查询
- `flash-sale-seckill-service`：秒杀资格尝试、结果查询

以下服务目前**只有工程骨架或路由占位**，还没有业务控制器接口，不纳入本文：

- `flash-sale-order-service`
- `flash-sale-payment-service`

## 2. 访问说明

### 2.1 推荐访问入口

推荐统一通过 Gateway 访问：

- Gateway 地址：`http://localhost:18080`
- 用户服务路由前缀：`/api/users`

完整示例：

- `POST http://localhost:18080/api/users/register`
- `POST http://localhost:18080/api/users/login`
- `GET http://localhost:18080/api/users/me`
- `POST http://localhost:18080/api/activities`
- `POST http://localhost:18080/api/activities/{activityId}/publish`
- `POST http://localhost:18080/api/activities/{activityId}/codes/import`
- `GET http://localhost:18080/api/activities/{activityId}/codes/import-batches`
- `DELETE http://localhost:18080/api/activities/{activityId}`
- `GET http://localhost:18080/api/public/activities`
- `GET http://localhost:18080/api/public/activities/{activityId}`
- `POST http://localhost:18080/api/seckill/activities/{activityId}/attempt`
- `GET http://localhost:18080/api/seckill/results/{activityId}`

### 2.2 鉴权规则

- `POST /api/users/register`：公开接口，无需登录
- `POST /api/users/login`：公开接口，无需登录
- `GET /api/public/activities`：公开接口，无需登录
- `GET /api/public/activities/{activityId}`：公开接口，无需登录
- `GET /api/users/me`：需携带 `Authorization: Bearer <accessToken>`
- `GET /api/users/{userId}`：需携带 `Authorization: Bearer <accessToken>`，且角色必须为 `ADMIN` 或 `PUBLISHER`
- `/api/activities/**`：需携带 `Authorization: Bearer <accessToken>`，且角色必须为 `ADMIN` 或 `PUBLISHER`
- `/api/seckill/**`：需携带 `Authorization: Bearer <accessToken>`

### 2.3 请求头约定

建议所有请求都带上：

- `Content-Type: application/json`
- `X-Request-Id: <唯一请求号>`

特殊说明：

- `POST /api/activities/{activityId}/codes/import` 为文件上传接口，请使用 `multipart/form-data`
- 文件字段名固定为 `file`

说明：

- 当前系统**不会自动生成** `X-Request-Id`
- 如果请求带了 `X-Request-Id`，响应体中的 `requestId` 会原样回传
- 如果未传，响应体中的 `requestId` 为 `null`
- `POST /api/seckill/activities/{activityId}/attempt` 当前会强制校验 `X-Request-Id`

### 2.4 直连服务说明

如果不走 Gateway，直接访问 `flash-sale-user-service`（默认端口 `9001`），则：

- `register`、`login` 仍可直接调用
- `me`、`/{userId}` 不会自动解析 `Bearer Token`
- 这两个接口依赖上游透传的用户头：
  - `X-User-Id`
  - `X-Username`
  - `X-Role`

因此联调时建议统一走 Gateway。

## 3. 统一响应格式

所有已实现接口都返回统一结构：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "requestId": "REQ-20260415-0001",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | string | 业务码，成功时固定为 `SUCCESS` |
| `message` | string | 响应描述，成功时固定为 `ok` |
| `requestId` | string/null | 请求头 `X-Request-Id` 的透传值 |
| `data` | object/null | 业务数据 |

## 4. 通用错误码

当前代码中已经明确实现的错误码如下：

| HTTP 状态码 | 业务码 | 含义 |
| --- | --- | --- |
| `400` | `INVALID_ARGUMENT` | 参数非法、用户名重复、用户名或密码错误、用户不存在、用户被禁用 |
| `401` | `UNAUTHORIZED` | 未登录、缺少 Bearer Token、Token 无效或已过期 |
| `403` | `FORBIDDEN` | 当前角色无权访问资源 |
| `500` | `SYSTEM_ERROR` | 未被显式处理的服务端异常 |

错误响应示例：

```json
{
  "code": "UNAUTHORIZED",
  "message": "缺少有效的 Bearer Token",
  "requestId": "REQ-20260415-0002",
  "data": null
}
```

## 7. 接口详情

### 7.1 用户注册

- 方法：`POST`
- 路径：`/api/users/register`
- 鉴权：否

请求体：

| 字段 | 类型 | 必填 | 约束 | 说明 |
| --- | --- | --- | --- | --- |
| `username` | string | 是 | 非空，最大 64 位 | 用户名 |
| `password` | string | 是 | 非空，长度 8 到 64 位 | 登录密码 |
| `nickname` | string | 否 | 最大 64 位 | 昵称 |
| `phone` | string | 否 | 最大 32 位 | 手机号 |

请求示例：

```json
{
  "username": "buyer_new",
  "password": "FlashSale@123",
  "nickname": "抢购用户",
  "phone": "13800000001"
}
```

成功响应示例：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "requestId": "REQ-USER-001",
  "data": {
    "id": 4,
    "username": "buyer_new",
    "role": "USER",
    "status": "ENABLED",
    "nickname": "抢购用户",
    "phone": "13800000001"
  }
}
```

业务说明：

- 注册成功后默认角色为 `USER`
- 注册成功后默认状态为 `ENABLED`
- 密码入库前会使用 BCrypt 加密

常见失败场景：

| HTTP 状态码 | 业务码 | message |
| --- | --- | --- |
| `400` | `INVALID_ARGUMENT` | `用户名已存在` |

| 字段 | 类型 | 必填 | 说明 |

| `password` | string | 是 | 密码 |

请求示例：


```json
{
  "code": "SUCCESS",

  "requestId": "REQ-USER-002",
  "data": {

      "id": 3,
```

常见失败场景：

| --- | --- | --- |

### 7.5 创建活动

- 方法：`POST`

- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

请求体示例：

```json
{
  "title": "5元礼品卡秒杀",
  "description": "限量兑换",

  "totalStock": 100,
  "priceAmount": 0,
```

### 7.6 编辑活动

- 方法：`PUT`
- 路径：`/api/activities/{activityId}`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- 仅允许编辑 `UNPUBLISHED` 状态的活动
- 更新库存时会同步重置 `availableStock`

### 7.7 查询活动详情

- 方法：`GET`
- 路径：`/api/activities/{activityId}`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- 返回活动基础字段以及派生状态 `phase`
- `phase` 当前取值：`PREVIEW`、`ONGOING`、`ENDED`

### 7.8 后台活动列表

- 方法：`GET`
- 路径：`/api/activities`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- 当前返回所有未删除活动
- 返回结构为 `ActivitySummaryResponse` 数组

### 7.9 发布活动

- 方法：`POST`
- 路径：`/api/activities/{activityId}/publish`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- `publishMode=IMMEDIATE` 时会立即切换为 `PUBLISHED` 并预热 Redis 库存、活动详情和可见活动索引
- `publishMode=SCHEDULED` 且 `publishTime` 仍未到时，仅保留 `UNPUBLISHED` 状态，等待调度器到点发布
- `codeSourceMode=THIRD_PARTY_IMPORTED` 时会校验 `redeem_code` 表中的可用码量是否不少于活动库存

### 7.10 下线活动

- 方法：`POST`
- 路径：`/api/activities/{activityId}/offline`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- 下线后活动状态切换为 `OFFLINE`
- 会同步删除 Redis 库存 Key、活动详情 Key，并移除可见活动索引

### 7.11 删除活动

- 方法：`DELETE`
- 路径：`/api/activities/{activityId}`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`

业务说明：

- 删除采用逻辑删除，实际会将 `activity_product.is_deleted` 标记为 `1`
- 仅允许删除 `UNPUBLISHED` 或 `OFFLINE` 状态活动
- 删除后活动不会再出现在后台列表和详情查询中
- 删除时会同步清理 Redis 库存 Key、活动详情 Key 和可见活动索引

常见失败场景：

| HTTP 状态码 | 业务码 | message |
| --- | --- | --- |
| `400` | `INVALID_ARGUMENT` | `仅未发布或已下线活动允许删除` |
| `400` | `INVALID_ARGUMENT` | `活动不存在` |

### 7.12 导入第三方兑换码

- 方法：`POST`
- 路径：`/api/activities/{activityId}/codes/import`
- 鉴权：是
- 角色要求：`ADMIN` 或 `PUBLISHER`
- `Content-Type`：`multipart/form-data`

路径参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `activityId` | number | 是 | 活动 ID |

表单参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file` | file | 是 | 兑换码文件，仅支持 `csv`、`xlsx` |

请求示例：

```http
POST /api/activities/1001/codes/import HTTP/1.1
Authorization: Bearer <accessToken>
X-Request-Id: REQ-ACTIVITY-IMPORT-001
Content-Type: multipart/form-data
```

成功响应示例：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "requestId": "REQ-ACTIVITY-IMPORT-001",
  "data": {
    "batchNo": "IMP-1001-20260418153000001-1234",
    "fileName": "codes.csv",
    "totalCount": 5,
    "successCount": 2,
    "failedCount": 3,
    "failures": [
      {
        "lineNumber": 3,
        "rawCode": "CODE1001",
        "reason": "DUPLICATE_IN_FILE"
      },
      {
        "lineNumber": 4,
        "rawCode": "BAD CODE",
        "reason": "INVALID_FORMAT"
      },
      {
        "lineNumber": 5,
        "rawCode": "",
        "reason": "EMPTY_CODE"
      }
    ]
  }
}
```

业务说明：

- 仅允许 `UNPUBLISHED` 且 `codeSourceMode=THIRD_PARTY_IMPORTED` 的活动导入兑换码
- 支持识别首行表头：`code`、`redeem_code`、`兑换码`
- 当前 CSV/XLSX 都按首列读取兑换码
- 成功导入的兑换码写入 `redeem_code` 表，状态固定为 `AVAILABLE`
- 失败行会按批次写入失败明细表，当前失败原因包括：
  - `EMPTY_CODE`
  - `INVALID_FORMAT`

### 7.17 秒杀资格尝试

- 方法：`POST`
- 路径：`/api/seckill/activities/{activityId}/attempt`
- 鉴权：是

请求头：

| Header | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 是 | `Bearer <token>` |
| `X-Request-Id` | 是 | 请求幂等标记 |

路径参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `activityId` | number | 是 | 活动 ID |

成功响应示例：

```json
{
  "code": "SECKILL_PROCESSING",
  "message": "请求已受理，请稍后查询结果",
  "requestId": "REQ-SECKILL-001",
  "data": {
    "activityId": 1002,
    "status": "PROCESSING"
  }
}
```

业务说明：

- 秒杀入口要求用户已登录，并由 Gateway 透传 `X-User-Id`、`X-Username`、`X-Role`
- 服务会先读取活动缓存并校验活动是否仍处于 `PUBLISHED`
- Lua 脚本在一次执行中完成时间窗校验、重复请求校验、限购校验和库存预扣
- Lua 校验通过后发送 `order.create` 消息，并写入 `seckill:result:{activityId}:{userId}` 为 `PROCESSING`
- 当前阶段仅打通秒杀入口最小闭环，`SUCCESS`、`PENDING_PAYMENT` 等最终状态将在后续订单/支付任务中回写

常见返回场景：

| HTTP 状态码 | 业务码 | message |
| --- | --- | --- |
| `200` | `SECKILL_PROCESSING` | `请求已受理，请稍后查询结果` |
| `200` | `ACTIVITY_NOT_STARTED` | `活动未开始` |
| `200` | `ACTIVITY_ENDED` | `活动已结束` |
| `200` | `ACTIVITY_OFFLINE` | `活动已下线或未发布` |
| `200` | `OUT_OF_STOCK` | `库存不足` |
| `200` | `DUPLICATE_REQUEST` | `重复请求` |
| `200` | `OVER_PURCHASE_LIMIT` | `超出限购` |
| `400` | `INVALID_ARGUMENT` | `缺少 X-Request-Id 请求头` |

### 7.18 秒杀结果查询

- 方法：`GET`
- 路径：`/api/seckill/results/{activityId}`
- 鉴权：是

路径参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `activityId` | number | 是 | 活动 ID |

成功响应示例：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "requestId": "REQ-SECKILL-002",
  "data": {
    "status": "SUCCESS",
    "orderNo": "SO202604190001",
    "message": "抢购成功",
    "code": "ABCDEF123456",
    "updatedAt": "2026-04-19T10:05:00"
  }
}
```

无结果时响应示例：

```json
{
  "code": "SUCCESS",
  "message": "ok",
  "requestId": "REQ-SECKILL-003",
  "data": {
    "status": "INIT",

  }

```

业务说明：

- 后续订单与支付任务会继续回写 `SUCCESS`、`FAIL`、`PENDING_PAYMENT` 等最终状态
