## 1. 文档目标

本文档用于把“功能需求与技术架构”继续细化为“可落地开发说明”。重点解决以下问题：

- 明确项目第一期实施范围，避免一次性铺太大。
- 统一服务、接口、数据库、缓存、消息的命名口径。
- 将业务流程拆解为可编码、可联调、可测试的模块设计。
- 补齐状态机、幂等、补偿、异常处理和验收标准。
- 给出推荐仓库结构、开发顺序和里程碑，方便直接进入实施阶段。

本文档默认面向单仓库、多模块、单机联调版本，不以生产级高可用部署为第一目标。

## 2. 实施范围与阶段边界

### 2.1 第一期实施目标

第一期建议完成以下 P0 闭环：

- 用户注册、登录、JWT 鉴权、角色识别。
- 活动创建、编辑、立即发布、定时发布、下线、详情查询。
- 第三方兑换码导入与批次校验。
- 秒杀资格校验、Redis 预扣库存、异步建单。
- 免费抢购发码闭环。
- 支付型活动待支付建单、模拟支付、支付成功发码闭环。
- 订单查询、兑换码查询、秒杀结果查询。
- 基础审计日志、异常告警日志、导出任务基础能力。

### 2.2 暂不纳入第一期

- 真实微信支付、支付宝支付。
- 多租户、结算、账务中心。
- 短信、站内信、邮件等真实通知渠道。
- 分布式事务框架、分库分表、集群高可用。
- 风控中心、验证码、设备指纹。

### 2.3 统一命名约定

需求文档中同时出现了 `activityId` 和 `product_id`。为避免实现阶段混乱，本文档统一采用以下约定：

- 接口层、缓存层、消息层统一使用 `activityId`。
- 数据库字段统一使用 `activity_id`。
- 表名保留 `activity_product`，表示“活动商品”实体。
- 所有订单都从属于一个 `activity_id`，不再新增 `product_id` 字段。

## 5. 服务详细设计

## 5.1 Gateway

### 5.1.1 核心职责

- 统一 API 入口。
- 解析 JWT，写入用户上下文 Header。
- 做基础 RBAC 鉴权。
- 记录请求日志与 `X-Request-Id`。
- 对秒杀入口做基础限流和黑名单预留。

### 5.1.2 关键规则

- 所有创建类接口要求携带 `X-Request-Id`。
- 未登录用户不可访问后台接口和秒杀接口。
- 网关只做基础身份校验，不承接业务逻辑。
- 用户信息透传字段建议包括 `X-User-Id`、`X-Role`、`X-Username`。

## 5.2 用户服务

### 5.2.1 业务边界

- 负责注册、登录、用户资料查询。
- 密码采用 BCrypt 存储。
- 角色字段只允许 `ADMIN`、`PUBLISHER`、`USER`。

### 5.2.2 核心接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/users/register` | 用户注册 |
| POST | `/api/users/login` | 用户登录 |
| GET | `/api/users/me` | 获取当前登录用户 |
| GET | `/api/users/{userId}` | 后台按 ID 查询用户 |

## 5.3 活动服务

### 5.3.1 业务边界

- 活动创建、编辑、查询、发布、下线。
- 库存初始化和缓存预热。
- 定时发布调度。
- 第三方兑换码批次导入与校验。
- 对用户侧活动列表按 `publishMode + publishTime` 过滤可见性。

### 5.3.2 内部模块建议

- `activity-domain`：活动实体、状态计算、发布校验。
- `activity-cache`：活动详情缓存、库存预热。
- `code-import`：导入解析、重复校验、批次落库。
- `publish-scheduler`：定时发布调度。

### 5.3.3 发布前校验清单

- 活动标题、库存、时间窗、支付模式、限购规则完整。
- `publishTime <= startTime`。
- 活动总库存大于 0。
- 第三方导入模式下可用兑换码数量不少于活动库存。
- 系统生成模式下不校验码池数量，但必须开启唯一编码规则。

## 5.4 秒杀服务

### 5.4.1 业务边界

- 唯一高并发秒杀入口。
- 不允许直接写 MySQL。
- 秒杀请求只做资格校验、Redis 原子扣减、消息投递、结果返回。

### 5.4.2 入口处理流程

1. 校验 `X-Request-Id`。
2. 校验活动是否对用户可见。
3. 执行 Redis Lua：
   - 活动时间校验
   - 重复请求校验
   - 限购校验
   - 库存扣减
4. Lua 成功后发送 `order.create` 消息。
5. 将结果写为 `PROCESSING`，快速返回。
6. 用户通过轮询接口查询最终状态。

### 5.4.3 返回码建议

| 业务码 | 含义 |
| --- | --- |
| `SECKILL_PROCESSING` | 请求已受理，等待异步结果 |
| `ACTIVITY_NOT_STARTED` | 活动未开始 |
| `ACTIVITY_ENDED` | 活动已结束 |
| `ACTIVITY_OFFLINE` | 活动已下线或未发布 |
| `OUT_OF_STOCK` | 库存不足 |
| `DUPLICATE_REQUEST` | 重复请求 |
| `OVER_PURCHASE_LIMIT` | 超出限购 |

## 5.5 订单兑换码服务

### 5.5.1 业务边界

- 消费秒杀成功消息并建单。
- 维护订单状态、支付状态、发码状态。
- 按活动来源分配第三方兑换码或生成系统兑换码。
- 处理库存回补、限购回补和异常告警。
- 负责导出任务创建和执行。

### 5.5.2 核心分层

- `order-app`：建单、状态流转、查询。
- `code-app`：兑换码分配、兑换码生成、发码记录。
- `compensation-app`：失败补偿、死信处理。
- `export-app`：导出任务和文件生成。

### 5.5.3 关键规则

- 免费活动建单成功后立即发码。
- 支付活动先创建待支付订单，支付成功后发码。
- 第三方导入码不足时不得创建成功订单。
- 系统生成码失败进入异常发码状态，不回退到第三方码模式。

## 5.6 支付服务

### 5.6.1 业务边界

- 创建模拟支付单。
- 记录支付流水。
- 发送支付成功消息。
- 发送超时关单消息。

### 5.6.2 关键规则

- 支付服务不直接调用订单服务发码。
- 支付回调使用 `transactionNo` 做幂等。
- 超时关闭只处理仍处于 `WAIT_PAY` 的订单。

## 7. Redis 详细设计

## 7.1 Key 设计

| Key | 类型 | TTL | 说明 |
| --- | --- | --- | --- |
| `seckill:stock:{activityId}` | String | 活动结束后 24h | 秒杀库存 |
| `seckill:limit:{activityId}:{userId}` | String | 活动结束后 24h | 用户购买次数 |
| `seckill:req:{activityId}:{userId}:{requestId}` | String | 10 min | 请求幂等标记 |
| `seckill:result:{activityId}:{userId}` | Hash | 活动结束后 24h | 抢购结果 |
| `activity:detail:{activityId}` | Hash | 活动结束后 24h | 活动详情 |
| `activity:visible:list` | ZSet | 长期 | 用户可见活动索引，score 为 `publishTime` |

## 7.2 Lua 脚本设计

### 7.2.1 输入

- `KEYS[1] = seckill:stock:{activityId}`
- `KEYS[2] = seckill:limit:{activityId}:{userId}`
- `KEYS[3] = seckill:req:{activityId}:{userId}:{requestId}`
- `ARGV[1] = nowEpochMillis`
- `ARGV[2] = startEpochMillis`
- `ARGV[3] = endEpochMillis`
- `ARGV[4] = purchaseLimitCount`
- `ARGV[5] = requestMarkerTtlSeconds`

### 7.2.2 返回码

| 返回值 | 说明 |
| --- | --- |
| `0` | 校验成功并已扣减库存 |
| `1` | 活动未开始 |
| `2` | 活动已结束 |
| `3` | 库存不足 |
| `4` | 重复请求 |
| `5` | 超出限购 |

### 7.2.3 脚本约束

- 一次执行完成时间校验、重复请求校验、限购校验和库存扣减。
- 成功后必须写入请求标记，防止客户端重放。
- 失败不应改动库存。
- 限购计数只在成功预扣库存时加 1。

## 7.3 抢购结果缓存结构

建议使用 Hash：

| 字段 | 说明 |
| --- | --- |
| `status` | `PROCESSING/SUCCESS/FAIL/PENDING_PAYMENT` |
| `orderNo` | 订单号 |
| `message` | 结果描述 |
| `code` | 兑换码，未发码时为空 |
| `updatedAt` | 最近更新时间 |

## 8. RabbitMQ 详细设计

## 8.1 Exchange 与队列

统一采用 `topic exchange`，建议命名如下：

| Exchange | Routing Key | Queue | 生产者 | 消费者 |
| --- | --- | --- | --- | --- |
| `flash.sale.event.exchange` | `order.create` | `flash.sale.order.create.queue` | 秒杀服务 | 订单服务 |
| `flash.sale.event.exchange` | `payment.success` | `flash.sale.payment.success.queue` | 支付服务 | 订单服务 |
| `flash.sale.event.exchange` | `order.timeout.close` | `flash.sale.order.timeout.close.queue` | 支付服务 | 订单服务 |
| `flash.sale.event.exchange` | `export.generate` | `flash.sale.export.generate.queue` | 订单服务 | 订单服务 |

## 8.2 统一消息模型

```json
{
  "messageId": "9f97d055-8fef-4eeb-a2b5-73f0bb420001",
  "eventType": "order.create",
  "bizKey": "activity:1001:user:2001:req:REQ001",
  "occurTime": "2026-04-14T11:00:00",
  "payload": {
    "activityId": 1001,
    "userId": 2001,
    "requestId": "REQ001",
    "needPayment": false,
    "codeSourceMode": "THIRD_PARTY_IMPORTED"
  }

```

## 8.3 消费规则

- 所有消费者先做幂等校验，再执行业务处理。
- 业务处理成功后再 ACK。
- 可重试异常进入有限次重试。
- 超过阈值进入死信队列并落异常记录。
- 死信队列中的订单类消息必须支持人工补偿。

## 9. 状态机设计

## 9.1 活动状态

实现建议采用“发布状态 + 时间窗派生状态”双层模型：

| 维度 | 枚举值 | 说明 |
| --- | --- | --- |
| `publish_status` | `UNPUBLISHED` | 未发布，对普通用户不可见 |
| `publish_status` | `PUBLISHED` | 已发布，对普通用户可见 |
| `publish_status` | `OFFLINE` | 已下线，不可参与 |
| `phase` | `PREVIEW` | 已发布但 `now < startTime` |
| `phase` | `ONGOING` | `startTime <= now <= endTime` |
| `phase` | `ENDED` | `now > endTime` |

## 9.2 订单状态

| 字段 | 枚举值 | 说明 |
| --- | --- | --- |
| `order_status` | `INIT` | 刚建单 |
| `order_status` | `CONFIRMED` | 业务成功闭环完成 |
| `order_status` | `CLOSED` | 超时关闭或人工关闭 |
| `order_status` | `FAILED` | 建单或发码失败 |
| `pay_status` | `NO_NEED` | 免费活动 |
| `pay_status` | `WAIT_PAY` | 待支付 |
| `pay_status` | `PAID` | 已支付 |
| `pay_status` | `CLOSED` | 支付关闭 |
| `code_status` | `PENDING` | 待发码 |
| `code_status` | `ISSUED` | 已发码 |
| `code_status` | `FAILED` | 发码失败 |
| `code_status` | `NOT_REQUIRED` | 无需发码，仅作保留 |

## 9.3 典型状态流转

### 免费抢购

`INIT -> CONFIRMED`  
`PENDING -> ISSUED`

### 支付抢购

`INIT + WAIT_PAY + PENDING -> PAID -> ISSUED -> CONFIRMED`

### 支付超时

`INIT + WAIT_PAY -> CLOSED`

### 发码异常

`INIT/PAID -> code_status=FAILED -> order_status=FAILED`

## 10. 核心链路与补偿设计

## 10.1 免费抢购链路

1. 秒杀服务执行 Lua 校验成功。
2. 秒杀服务发送 `order.create`。
3. 订单服务建单。
4. 若为第三方导入码，锁定一条 `AVAILABLE` 兑换码并改为 `ASSIGNED`。
5. 若为系统生成码，生成唯一编码并落库。
6. 更新订单 `code_status=ISSUED`、`order_status=CONFIRMED`。
7. 写回 Redis 结果为 `SUCCESS`。

失败补偿：

- 建单失败：回补 Redis 库存和限购计数。
- 导入码不足：回补 Redis 库存和限购计数，结果置为 `FAIL`。
- 系统生成码失败：订单置 `FAILED`，写异常日志，不执行来源切换。

## 10.2 支付抢购链路

1. 秒杀成功后订单服务创建 `WAIT_PAY` 订单。
2. 结果缓存写为 `PENDING_PAYMENT`。
3. 用户发起模拟支付，支付服务落支付记录。
4. 支付回调或模拟成功后发送 `payment.success`。
5. 订单服务消费后更新 `pay_status=PAID`。
6. 执行兑换码分配或生成。
7. 成功后写回 `SUCCESS`。

失败补偿：

- 支付成功消息重复：按 `transactionNo` 和 `orderNo` 幂等。
- 支付后发码失败：订单进入异常待处理，写告警日志。
- 支付超时关闭：回补 Redis 库存和限购计数。

## 10.3 库存回补原则

允许回补的场景：

- `order.create` 最终失败。
- 第三方导入码不足导致建单失败。
- 待支付订单超时关闭。

不自动回补的场景：

- 已支付但发码失败。
- 已成功发码后的后续异常。

## 14. 测试与验收设计

## 14.1 功能测试清单

- 立即发布活动创建成功。
- 定时发布活动在 `publishTime` 前不可见，到点后可见。
- 第三方导入码数量不足时禁止发布。
- 免费活动抢购成功后可查询订单和兑换码。
- 支付活动支付成功后才发码。
- 超时未支付订单被关闭并回补库存。

## 14.2 并发测试清单

- 同一活动并发抢购不出现负库存。
- 同一用户重复请求不出现重复建单。
- 一人一买场景下不出现多单成功。
- 支付重复回调不出现重复发码。
- 第三方导入码不会被分配给多个订单。

## 14.3 验收指标

| 类别 | 指标 |
| --- | --- |
| 功能闭环 | 免费抢购和支付抢购都能完整跑通 |
| 正确性 | 不超卖、不重复下单、不重复发码 |
| 性能 | 单活动秒杀接口本地目标 `100~300 QPS` |
| 一致性 | Redis、MQ、MySQL 最终状态一致 |
| 可观测性 | 能定位请求、订单、支付、发码和导出链路 |

## 15. 风险与待确认项

## 15.1 已知实现风险

- 单机联调环境下 Redis、RabbitMQ、MySQL 任意一个异常都会打断闭环。
- 定时发布依赖本机时钟与调度扫描频率，容易出现边界偏差。
- 系统生成码规则如果过于简单，可能在并发下出现唯一性冲突。
- 第三方导入码质量差会直接影响发码成功率。

## 15.2 实施前建议确认

- 兑换码格式规则是否固定，是否区分不同商家模板。
- 支付超时时间是否统一，例如固定 15 分钟。
- 导出文件是否需要权限签名链接或仅本地静态目录即可。
- 是否需要活动审核流，第一期文档暂未纳入。

## 16. 结论

本实施文档的核心作用不是重复需求说明，而是把现有方案落成一套可以直接开工的实现基线。建议你在实际编码时，以 M0 到 M6 的顺序推进，并始终围绕两条主链路验收：

- 免费抢购：活动发布 -> 秒杀成功 -> 建单 -> 发码 -> 查询结果
- 支付抢购：活动发布 -> 秒杀成功 -> 待支付 -> 支付成功 -> 发码 -> 查询结果

只要这两条链路在“第三方导入码”和“系统生成码”两种模式下都能跑通，并且库存、幂等、补偿没有问题，这个项目就具备了可答辩、可演示、可继续扩展的实施基础。
